# 角色设计指南

> 本文件是鸣潮抽卡模拟器做单个角色时的**设计指南** —— 不是代码指南。
> 代码路径、文件结构、tooltip HTML 标签等实装细节见 [../../CLAUDE.md](../../CLAUDE.md)。
> 本指南只讲**怎么把一个角色从官方资料读出来、做出模拟器取舍、写成设计文档**。
>
> 内容来源：本次亲做角色总结会（2026-06-27，纪要见 [characters/会议纪要-亲做角色设计总结.md](characters/会议纪要-亲做角色设计总结.md)），覆盖守岸人 / 忌炎 / 安可 / 卡提希娅四个亲做角色。旧版 [characters/README.md](characters/README.md) 的移植分级表、文案铁律、自检清单仍然有效，但被本指南吸收并补强。

## 一、模拟器的定位 · 决定一切取舍

这个项目叫**鸣潮抽卡模拟器**。主题是抽卡，战斗是附带的——战斗存在只是为了让抽出来的强力角色和武器有相应体验，**不是要把鸣潮的战斗系统完整复刻**。

这条定位决定了三件事：

1. **官方数据是输入，不是目标**：官方面板/技能/链是设计师的素材，不是我们要 1:1 还原的对象
2. **战斗必须简化**：原版是动作 ACT，我们是 AP 回合制。**一角色一核心 + 一条 FORTE 资源条** —— 不要塞 3 个独立子系统
3. **取舍要做减法**：复杂角色（如卡提希娅）"给同等的尊重" 不是"全盘照搬"，而是"花一样的力气读官方，再做减法"

## 二、五层移植流程（顺序不可跳过）

每做一个角色按下面五层走，**前一层没敲定，不进下一层**。这条流程是本次会议的主要成果，取代旧"强制五步工作流"。

### 第 1 层 · 官方数据搬运

直接抄 `docs/sources/characters/individual/<角色名>.json` 的 `lv90_stats`、技能列表、6 条共鸣链。这一层**没有"设计"成分**，只忠实搬运。

值得留一句的是：面板里**已经藏着答案** —— 比如守岸人 ATK 288 比主 C 主流 400+ 低一档，暗示她不是输出位；卡提希娅 HP 14800 比主 C 主流 12000 高一档、ATK 312 被刻意压低，暗示她是 HP 核角色。**先把面板读出声来**，再读技能。

### 第 2 层 · 官方资料分析 · 读出设计师意图

**起点不是"我要怎么改"，而是"官方在做什么"。** 我们不是凭空设计，是从官方资料里读懂设计师的心，再去想怎么在一个回合制简化版里复刻那个意图。这一层按住自己 —— **先不要想回合制怎么改，先理解原版**。

四件事做下来：

1. **读技能描述**：常态攻击 / 共鸣技能 / 共鸣解放 / 共鸣回路 / 变奏 / 延奏 / 固有技能逐条读。不是列出来交差，是读懂"官方想让她在战场上做什么"
2. **读共鸣链文案**：6 条链逐条读，看官方把什么加成放在哪条链 —— 加在 1 链的通常是"循环顺畅化"，加在 6 链的通常是"质变爆发"。**链的位置不是随意的，是设计师对强度曲线的安排**
3. **读懂原版循环**：把上面两步合起来，看这个角色在动作游戏里的循环节奏是怎样的 —— 起手 → 攒什么资源 → 放什么招 → 切人 or 继续 → 何时爆发
4. **提炼"官方想表达的她"**：用一两句自然语言写出这个角色的设计魂。这一句话就锁住了核心 —— 后面所有取舍都围绕这一句话

**复杂角色专用方法**：当机制名词密密麻麻（如卡提希娅的决意/风蚀/芙露德莉斯/剑之影/集意/显化/双阶段解放…）时，先用一句话**抓主循环骨架**，再列一张"每个机制名词在骨架里起什么作用"的表 —— 标出哪些是核心、哪些可舍弃。骨架不确定就先别取舍。

### 第 3 层 · 模拟器取舍设计 · 核心设计区

读懂官方之后才开始折算。取舍有五条原则：

1. **核心一个**：每个角色只保留一个核心机制，6 条共鸣链全部围绕这一个核心加成。守岸人=星域、忌炎=锐意之势、安可=失序/黑咩、卡提希娅=决意/芙露 —— 链不对核心就被视为设计偏差
2. **回合制折算**：原版秒数 → 回合（6-12 秒 ≈ 2 回合，14-24 秒 ≈ 3 回合，25-30 秒 ≈ 4 回合，超长 CD 写"每场战斗 1 次"）
3. **合并同类项**：动作游戏特有的细腻机制（空军/空中攻击/分身/多段派生/范围/追击距离）按"模拟器抽象"合并到一两个 buff 或一次结算里
4. **战斗不是核心**：一个角色不要塞 3 个独立子系统，能合并到 FORTE 一条资源条解决就不要开第二条
5. **取舍必须明示**：哪条官方机制我们不做、为什么不做（太动作 / 与回合制不兼容 / 强度溢出 / 与战斗不是核心的定位冲突）—— 要在设计文档里讲清

**HP 核角色通用规则**（从卡提希娅升格为通用规则）：

> HP 核角色（伤害按 HP 倍率计算）的普攻/技能/重击倍率必须按 HP/ATK 倍数比下调，**不能从 ATK 核角色直接平移倍率**。
>
> 校准基线：HP/ATK ≈ 8.7×。让普攻 HP × 12% 折合 ATK × 100%，技能 HP × 22% 折合 ATK × 190%，重击 HP × 26% 折合 ATK × 225%，大招保留 HP × 46% 是 HP 角色该有的"天花板倍率"。
>
> 反例：卡提希娅直接平移 ATK 核的普攻 100%/技能 180%/重击 220%，结果普攻 ≈ 14800 伤害反超大招 ≈ 6800 伤害 2 倍，机制错乱。

### 第 4 层 · 文案与 UI 适配

- 专用名词（资源条、形态、招式名）加入 `src/ui/terms.js` 的 `TERM_DICT`
- 共鸣链文案中出现的术语同时加入 `render.js` 的 `CHAIN_TERM_PATTERNS`，使术语在共鸣链和技能描述里都能悬停
- 时间统一用回合，不写秒
- 文案铁律见下方"前端文案规范"小节

### 第 5 层 · 写设计文档

设计师环节，不是代码环节。要求见下方"设计文档怎么写"小节。

## 三、移植分级策略 · 一角色一档实装深度

不是所有角色都用守岸人级别（专属状态机）实装 —— 那是 8000+ 行新代码 + 高错率的事。按重要程度分四级，**逐级降级是诚实，不是偷懒**：

| 级别 | 文件改动 | 完成度 | 适用角色 |
|---|---|---|---|
| **S 级**（专属状态机）| chains.js + combat.js + chains-extracted.json + render.js customLines + terms.js | 结构化战斗 effect + 公式 tooltip + 重写 6 链 + 专属 combat helper | 真 SS/S 级核心 / 独特机制不可用工厂表达者，≈ 8~10 人 |
| **A 级**（工厂完整）| chains-extracted.json + chains.js 标准 effect + render.js `makeSkillLines` 配置 + terms.js | 公式 tooltip + 官方口吻链文案 + 标准 effect；不写专属 combat helper | 大多数限定 / 常驻强角 / 玩法能用通用 AP 模型表达者，≈ 20+ 人 |
| **B 级**（最小化）| chains-extracted.json（官方标题 + 简洁染色）+ SKILL_HINTS 简单 intro/normal/skill/burst | 文案染色 + 术语 tooltip；技能区只有简介 | 4★ 边缘 / 不熟的新角色 / 3.x 后期暂不深入者 |
| **C 级**（只收录）| 不新增机制，只保留 seq.js 原文或 fallback | 可以抽到和升级，但不承诺真实机制 | 明显低强度或资料不足角色 |

**升降级原则**：

- 默认 A 级工厂完整，**不要动不动升 S**
- A→S 只在"这个角色的核心决策无法用 makeSkillLines + 标准 effect 表达"时升
- **S 级不是奖励强度，而是奖励"机制必要性"**：守岸人星域、忌炎锐意这种才值得专属 helper
- B 级**不是凑数**，是承认暂时不懂这个角色的真实玩法。**瞎编 customLines 比裸数文案+染色更糟**
- 数值不熟就保守 → "宁可保守，不要瞎加倍率"

对照本指南实装的角色：守岸人、忌炎、卡提希娅都做 S 级；安可（状态机）是 S 级；今汐/长离/折枝/相里要/椿/珂莱塔/洛可可/菲比/布兰特/坎特蕾拉/嘉贝莉娜全员限定都用 A 级工厂；4★ 多用 A 级工厂或 B/C 级。

## 四、前端文案规范 · 玩家空间 vs 开发者空间

这条规范在 CLAUDE.md 已有简版，这里给完整版。**违反任何一条都是严重错误**。

### 1. intro 行只写身份

格式：`元素 · 武器 · 定位 · 「核心机制名」`。

- ❌ `「决意叠层 → 双形态 · 风蚀爆发」`（→ 是开发者速记）
- ❌ `「展开星域让主 C 越打越痛，是主输出窗口的保护者」`（替玩家分析强度）
- ✅ `气动 · 迅刀 · 限定主 C · 「决意」与「芙露德莉斯」双形态`
- ✅ `衍射 · 音感仪 · 限位辅助 · 「星域」`

### 2. 技能名必须用官方原文

从 `docs/sources/characters/individual/<角色名>.json` 的 `skills[].name` 取，禁止自己编造。

### 3. 共鸣链名称和效果来自官方 JSON

`chains[].name` 取名称，`chains[].desc` 理解效果。模拟器数值有意下调时，设计文档里必须标"官方原值 X → 模拟器下调至 Y，原因：B-Tier 数值约束"。

### 4. 术语归属要准确

通用游戏机制（如风蚀效应是气动属性效果）不标某个角色专属（如"卡提希娅 DoT"）。风蚀是鸣潮全游戏的"异常效应"通用体系，不是卡提希娅发明的。

### 4b. 模拟器无声骸技能作为独立伤害类型（铁律）

**模拟器不实装"声骸技能"作为独立的伤害类型/乘区**。这是声骸系统移植之初就定的规矩（见 [mechanisms/echo-system.md:81](mechanisms/echo-system.md)："核心原则：不实现声骸技能（取消主动释放）"）。

鸣潮官方的"声骸技能"是装备声骸后释放的主动技能。模拟器只保留**声骸套装效果**（套装 2/5 件加成、主词条、副词条），声骸本身不是主动技能。因此：

- **禁止**在角色设计文档、共鸣链文案、terms.js、chainEffects.js、技能 tooltip 中出现"声骸技能伤害""视为声骸技能伤害""声骸技能时全队..."等把"声骸技能"当作独立伤害类型或触发事件的表述
- 官方原文里写"视为声骸技能伤害"的招式，移植时归到 normal/skill/heavy/burst 四类之一（按招式动作类型判定），文档里写明归类结果
- 官方"施放声骸技能时触发 X"的共鸣链，移植时把触发条件换成模拟器里实际存在的招式（如"施放谱曲终末时"/"施放共鸣解放时"/"施放共鸣技能时"），并核对触发条件与实装一致
- "声骸套装 5 件效果"是另一回事——那是装备系统的套装加成，**保留**，不算"声骸技能"

**已发生过的失误**（2026-07）：弗洛洛、仇远、嘉贝莉娜、莫特斐、西格莉卡的设计文档和 seq.js 文案反复写"视为声骸技能伤害""声骸技能伤害 +X%"，但代码里 `dmgType` 只有 normal/skill/heavy/burst 四类，从未实装 `echoSkill` 类型。导致 chainEffects 里"声骸技能伤害 +80%"用 `skillDmg` 兜底，对 `dmgType: 'heavy'` 的谱曲终末完全失效。这是设计文档与实装长期偏离的持续性失误，根因是规矩没写进指南。

### 5. 时间统一用回合

6-12 秒 ≈ 2 回合；14-24 秒 ≈ 3 回合；25-30 秒 ≈ 4 回合；特殊长 CD（10 分钟）写"每场战斗 1 次"。

### 6. 不用分析性语句

不写"全部价值集中于""不依赖专属资源""这是主输出窗口"等替玩家分析的语言。**只陈述事实**。

### 7. 新术语同时加入 `TERM_DICT` 和 `CHAIN_TERM_PATTERNS`

确保在共鸣链和技能描述里都能悬停。

### 8. 文案 = 具体数值，tooltip = 公式（铁律）

- ❌ "对目标造成 100% 攻击力的衍射伤害"
- ✅ "对目标造成 309 点衍射伤害"，悬停 309 看公式 `= 攻击 309 × 100%`
- 数值随角色面板 / 共鸣链动态计算，文案动态生成
- tooltip 用 `<span class="tip" data-tip='公式'>`（虚线下划线）
- **凡是出现数字的地方都要有 tooltip 公式**，否则会"露出来"显得没做完

### 9. 玩家空间严禁出现开发者速记

| 速记 | 写法 |
|---|---|
| ❌ `→` | 写成"并"或"且" |
| ❌ `+` | 写成"加成"或"提升" |
| ❌ `buff` / `debuff` | 写成"增益"/"减益" |
| ❌ `core` / `核心` | 写成"核心机制"或具体机制名 |
| ❌ `叠层` / `叠层 N 层` | 写成"每叠 1 层 N%" |

### 10. 重击是 opt-in，不是 opt-out

- 需要重击的角色才加 `hasHeavy: true` flag
- 默认所有角色都没有重击，避免误伤
- 战斗 UI 同步：根据 `cur.hasHeavy` 动态布局 —— 有重击 4 列、无重击 3 列（重击按钮整列移除，不留灰按钮）；面板顶部 banner 的"重击 2AP/CD1"也按 hasHeavy 动态显隐

## 五、设计文档怎么写 · 设计师思维

设计文档是给设计看的，**不是给代码看的**。已有的守岸人/忌炎/安可/卡提希娅四份文档都存在代码思维残留（`effect key` 标识符、`实装位置` 列、表格化 `模拟器舍弃/合并`、`✅ 已实装`勾选框）—— 这是后续要逐步淘汰的格式。

### 写文档的五条原则

1. **设计者思维，不是代码思维**：用自然语言把自己当成这个角色的设计师、创作师、游戏师，把"她是一个什么样的角色、循环节奏是什么、爽点在哪、玩家抽到她会怎么用"讲清楚
2. **用玩家的话叙述**：不是写 `effect: 'fieldExtend', value: 2`，而是写"星域展开后，时间从 3 回合延长到 5 回合，且切人时星域不再消失"
3. **考虑越详细，bug 越少**：进入条件、退出条件、切人 carrying、与既有 buff 的叠加规则、与致命伤的关系、与回合结束的关系——这些边界都要写到。**写到的边界比代码里可能踩到的 bug 多一倍**
4. **官方数据搬运要带完整公式**：抄 `docs/sources/characters/individual/<角色名>.json` 时，技能描述和共鸣链描述必须**原样搬运完整文本**，包括所有伤害计算公式、系数、持续秒数、冷却秒数——不是只列名字和简短概要。这是后续取舍的"事实基础"，必须可追溯到官方原文
5. **最终设计必须完整重写一遍含公式**：做完模拟器取舍后，第 4 节"招式数值"和第 5 节"共鸣链 · 模拟器版"必须把**最终设计完成的技能和共鸣链效果完整重写一遍**，包含所有计算公式（倍率、AP、CD、伤害类型、持续时间）。表格可以列概览，但每个技能/链必须有一段完整文字描述，写清"这个招式 / 这条链在模拟器里最终是什么效果、伤害怎么算"。读者只看这一段就能完全理解模拟器版的实装行为

### 推荐章节结构

```markdown
# {角色名} 移植计划

> 状态 / 强度 Tier / 移植级别 / 定位（格式：`元素 · 武器 · 定位 · 「核心机制名」`）

## 1. 官方数据搬运
   —— 面板 + 技能完整描述（含官方伤害公式原文）+ 6 条共鸣链完整原文 + 面板信号解读

## 2. 官方资料分析 · 读懂官方想让她做什么
   —— 技能逐条读懂（不是罗列）
   —— 6 条共鸣链设计师意图分析（每条说明"为什么放在这里"）
   —— 一句话灵魂锁定

## 3. 模拟器取舍设计
   —— 保留的核心（4 件事以内）
   —— 舍弃/合并清单（明示为什么不做）
   —— 复杂角色给"机制名词在骨架里起什么作用表"
   —— HP 核角色补 HP/ATK 倍率校准

## 4. 招式数值 · 模拟器版完整设计
   —— 招式倍率表（含 AP 列）+ 资源系统参数
   —— 每个招式完整文字描述：触发条件 / 倍率 / 伤害类型 / CD / 与其他机制交互
   —— 数值与官方差异说明（如有下调）

## 5. 共鸣链 · 模拟器版完整设计
   —— 每条链：官方名称 + 模拟器作用（完整文字描述含公式）+ 触发条件 + 边界

## 6. 文案与 UI 适配
   —— 需要进 `TERM_DICT` 的术语列表
   —— intro 行（带正反例）

## 7. 设计边界（bug 防范点）
   —— 切人 carrying / 与既有 buff 叠加规则 / 致命伤交互 / 回合结束时机 / 字段命名与重置点
   —— 越细越好

## 8. 留给讨论的开放问题（如有）
```

### 写文档自检清单

- [ ] 技能名是否全部来自官方 JSON？
- [ ] 共鸣链名称和效果是否与官方 JSON 一致？

## 六、复杂角色做减法 · 卡提希娅范式

复杂角色（如卡提希娅，2.x 版本最终大 C，官方机制海）的处理范式：

1. **不被机制海淹没**：上来不要直接读 6 条链详情，先抓主循环骨架 —— 比如"卡提希娅攒决意 → 解放换形态 → 芙露形态攒风蚀攒新决意 → 解放按风蚀层数 HP 倍率收尾 → 退形态"四步
2. **每个机制名词列表归位**：决意、风蚀、芙露德莉斯、剑之影、集意、显化、双阶段解放… 每个名词在骨架里起什么作用，标"核心 / 可舍弃"。舍弃的从此不再讨论 —— 已经决定不做的机制不要再二次回锅
3. **一句话锁设计魂**：在所有取舍之前先锁住 —— "她是 HP 核双形态主 C，靠攒决意→变身芙露→叠风蚀→终结解放按层数 HP 倍率收尾此条循环，把一场战斗的爽点压在最后那一刀上"
4. **减法做减法**：模拟器给"同等的尊重 = 花一样力气读官方，再做减法"，不是"全盘照搬"。剑之影/空中召回/集意这种动作手感细节全删；6 类异常只取风蚀一种做主机制
5. **HP 核倍率校准**（通用规则，见第 3 层）

详细范例：`characters/会议纪要-亲做角色设计总结.md` 第三部分。

## 七、两个范例 · 守岸人 vs 卡提希娅

**守岸人**（典型限定辅助）：

- 设计魂："她是星域展开全队的治疗和增益者。她的脆皮主 C 在星域里既不会死又越打越准越打越痛。她的爽点不是自己爆发，是看着队友在自己铺的场地上爆发。"
- 复杂度：able 抽象优雅 —— 三段星域进阶合并为"开星域一次上全 buff"
- 详细：`characters/会议纪要-亲做角色设计总结.md` 第二部分

**卡提希娅**（典型复杂大 C）：

- 设计魂："她是 HP 核双形态主 C，靠攒决意 → 解放变身芙露德莉斯 → 芙露形态叠风蚀攒新决意 → 解放按风蚀层数 HP 倍率终结全场 四步循环，把爽点压在最后那一刀上。"
- 复杂度：高 —— 机制海做减法（剑之影/集意/空攻全删；6 异常只留风蚀）
- HP 核倍率校准：原版普攻 100% HP 反超大招 → 校准为 12% HP
- 详细：`characters/会议纪要-亲做角色设计总结.md` 第三部分

后续做新角色时，**先想"她更像守岸人还是更像卡提希娅"**，对号入座找对应范式。

## 九、工作纪律 · AI 必须遵守

违反任意一条的都是严重错误，不是小疏漏。

1. **每次提交必须署名**：commit message 末尾必须含 `Co-Authored-By: Claude <noreply@anthropic.com>`（或实际作者）
2. **已实装角色的设计决策不可擅自修改**：见官方数据和代码不一致时，记录差异，报告用户，等待决定。严禁提议删除用户的设计决策（如某条链的倍率、某个自定义折算系数）。严禁因"贴近官方"擅自改数值公式。架构优化时严禁顺手改角色行为和数值
3. **设计文档 > 官方数据**：当设计文档、官方数据、当前代码三方冲突时，优先级是：① 用户口头指示 → ② 设计文档（`docs/plans/`）→ ③ 当前实装代码 → ④ 官方数据（`docs/sources/`，最低）
4. **禁止再犯的错误**：
   - ❌ 看官方说 HP 公式就擅自把已实装的 ATK 改成 HP —— 越权
   - ❌ 看官方说 1 链不加治疗倍率就提议删 ×2.5 —— 越权
   - ❌ 重构代码时顺手改角色数值 —— 违反架构纪律
   - ❌ 写的设计文档共鸣链和实装代码完全对不上 —— 文档虚假
